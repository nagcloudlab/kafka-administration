# Lab 02 — Monitoring Setup (JMX → Prometheus → Grafana)

**Audience:** Ops / SRE — sets up visible metrics before the failure-mode chapters.
**Goal:** Every broker exports JMX metrics via `jmx_prometheus_javaagent`. Prometheus scrapes them. Grafana displays a pre-provisioned dashboard tied to the scenarios in Chapters 1–4.

Do this **before** Lab-03 onward — the failure demos are much easier to run with a live dashboard on-screen.

---

## 1. Objective

By the end of this lab you will:

1. Attach the **jmx_prometheus_javaagent** to each Kafka broker so metrics are exposed on `/metrics` at ports 7071–7073.
2. Run **Prometheus** + **Grafana** in Docker, scraping those endpoints.
3. Auto-provision a **Kafka Admin Training** dashboard whose panels map 1:1 to Lab 3–4 takeaways (URP, UnderMinISR, offline partitions, unclean elections, etc.).
4. Verify metrics update live as you produce traffic to the cluster.

## 3. Layout

Everything for this lab lives under `Lab-02-Monitoring-Setup/`:

```
Lab-02-Monitoring-Setup/
├── README.md                         (this doc)
├── up.sh                             (bring stack up — detects host IP, generates prometheus.yml)
├── down.sh                           (bring stack down)
├── docker-compose.yml                (Prometheus + Grafana)
├── prometheus.yml.template           (source of truth; __HOST_IP__ placeholder)
├── prometheus.yml                    (generated by up.sh — do not edit)
├── kafka-jmx-config.yaml             (which JMX MBeans to export)
└── grafana-provisioning/
    ├── datasources/prometheus.yaml   (auto-provisioned datasource)
    └── dashboards/
        ├── provider.yaml             (dashboard loader)
        └── kafka-admin.json          (the dashboard itself)
```

The Kafka brokers themselves stay outside Docker — same `kafka-lab/cluster.sh` as Lab-01, invoked with the new `*-monitoring` commands.

> **Why the template + `up.sh` instead of just `docker compose up`:** On Docker Desktop for WSL2, `host.docker.internal` inside a container is hardcoded to Docker Desktop's internal gateway (192.168.65.254), which cannot route back to the WSL host where the JMX exporters listen. `up.sh` sidesteps this by detecting the real WSL host IP and substituting it into `prometheus.yml`.

## 4. Step 1 — Get the JMX Prometheus Exporter agent

From the repo root:

```bash
cd Lab-02-Monitoring-Setup

# ~600 KB. v1.0.1 works with Java 17+; v1.4.0 is fine too.
curl -L -o jmx_prometheus_javaagent.jar \
  https://repo1.maven.org/maven2/io/prometheus/jmx/jmx_prometheus_javaagent/1.0.1/jmx_prometheus_javaagent-1.0.1.jar

ls -lh jmx_prometheus_javaagent.jar
```

The JMX config (`kafka-jmx-config.yaml`) is already in this folder — it maps ~15 broker JMX MBeans to Prometheus metric names.

---

## 5. Step 2 — Start brokers with monitoring on

`cluster.sh` already knows how to attach the JMX agent — no patch needed. It looks up the jar and config in this folder by default (override with `JMX_EXPORTER_JAR` / `JMX_EXPORTER_CONFIG` env vars if you moved them).

### If the cluster is already up (from Lab-01), do a rolling restart:

```bash
cd ../kafka-lab

for id in 101 102 103; do
  ./cluster.sh stop-broker              $id
  sleep 3
  ./cluster.sh start-broker-monitoring  $id
  sleep 5
done
```

### If the cluster is stopped, start the whole thing with monitoring:

```bash
cd ../kafka-lab
./cluster.sh start-monitoring
```

Each start line now prints `starting broker N (port 909M, JMX exporter on 707M)...` — that's the confirmation the agent is attached.

If the jar/config paths are wrong you'll get an early `monitoring on but JMX jar missing: ...` before the broker starts. Fix the download / paths and retry.

## 6. Step 3 — Verify each broker exposes `/metrics`

```bash
for port in 7071 7072 7073; do
  echo "=== broker on JMX port $port ==="
  curl -s "http://localhost:$port/metrics" | grep -E '^kafka_' | head -5
done
```

Expected: metric lines like

```
kafka_broker_bytesinpersec 0.0
kafka_server_replica_manager_underreplicatedpartitions 0.0
kafka_controller_activecontrollercount 1.0
```

Sample counts:

```bash
for port in 7071 7072 7073; do
  echo -n "port $port kafka metric count: "
  curl -s "http://localhost:$port/metrics" | grep -c '^kafka_'
done
# Expected: ~40-60 metrics per broker
```

If any port returns empty or connection-refused, jump to the Gotchas section (§ 11).

## 7. Step 4 — Bring up Prometheus + Grafana

```bash
cd ../Lab-02-Monitoring-Setup
./up.sh
docker compose ps
```

`up.sh` does three things:
1. Detects the WSL host IP (or uses `HOST_IP=<addr> ./up.sh` if you want to override).
2. Regenerates `prometheus.yml` from `prometheus.yml.template` with that IP.
3. Runs `docker compose up -d`.

Expected: both `kafka-lab-prometheus` and `kafka-lab-grafana` in `Up` state, plus a printed URL summary at the end.

### Confirm Prometheus is scraping

Open **<http://localhost:9090/targets>** in a browser.

Expected: the `kafka` job shows 3 targets, all with `State: UP`. If any are `DOWN`, § 11.

## 9. Step 6 — Prove the dashboard reacts to real events

Generate live traffic and watch panels move:

```bash
# In one shell
cd ../kafka-lab
$KAFKA/kafka-topics.sh --bootstrap-server localhost:9092 \
  --create --topic monitoring-smoke --partitions 6 --replication-factor 3

$KAFKA/kafka-producer-perf-test.sh \
  --topic monitoring-smoke \
  --num-records 500000 --record-size 1000 --throughput 5000 \
  --producer-props bootstrap.servers=localhost:9092 acks=all
```

**On the dashboard (should update within 5–10 s):**
- **Bytes in per broker** — climbs to ~5 MB/s spread across brokers.
- **Bytes out per broker** — similar (replication traffic).
- **Request-handler idle %** — drops from ~100% to something lower.

Now stop broker 102 gracefully to trigger a rebalance:

```bash
./cluster.sh stop-broker 102
```

- **UnderReplicatedPartitions** — spikes for brokers 101 and 103 (they lead partitions whose ISR just shrunk).
- **Brokers up** — drops to 2 (Prometheus marks broker 102's target as DOWN).

Restart with monitoring still on:

```bash
./cluster.sh start-broker-monitoring 102
```

- URP returns to 0 within a few seconds.

Cleanup:

```bash
$KAFKA/kafka-topics.sh --bootstrap-server localhost:9092 --delete --topic monitoring-smoke
```

## 10. Panels ↔ Scenarios cheat sheet

Every panel is labelled with the scenario it teaches. Print or keep this open during the failure-mode labs.

| Panel | Scenario | What "healthy" looks like |
|-------|----------|---------------------------|
| Brokers up | 1.1 · Controlled shutdown | 3, all green |
| Offline partitions | 1.1 · SIGKILL side | 0. Non-zero = controller can't elect. |
| Active controllers | infra sanity | Exactly 1. 0 or 2+ = split brain / no controller. |
| Unclean elections total | 2.2 · Unclean election | Should stay 0. Any increment = a data-loss event happened. |
| UnderReplicatedPartitions | 1.3 · URP is not a page | Occasional flaps OK; sustained > 0 = investigate. |
| UnderMinIsrPartitionCount | 2.1 · Durability triangle | **The real page metric.** Any non-zero = producers with `acks=all` are erroring. |
| Bytes in / out per broker | infra sanity | Balanced across brokers. Skew = leader imbalance. |
| Request-handler idle % | 4.1 · Rebalance storm / 4.2 · Quotas | > 60% is healthy. < 30% sustained = CPU pressure. |

## 11. Sanity checks & gotchas

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| `curl :7071/metrics` connection refused | Broker not restarted with `start-broker-monitoring` | Re-run § 5; broker log's first few lines should mention `Loaded [...]jmx_prometheus_javaagent` |
| `monitoring on but JMX jar missing` at broker start | Jar not downloaded, or in a non-default path | Redo § 4, or `export JMX_EXPORTER_JAR=/abs/path/agent.jar` before the start command |
| Prometheus target DOWN for one broker | Only that broker is missing the agent | `ps -ef \| grep broker-N` — the java command should contain `-javaagent:...` |
| Prometheus targets all DOWN | `up.sh` detected the wrong host IP | Re-run `HOST_IP=<correct-addr> ./up.sh`. Find the reachable IP with `ip -4 addr` |
| `up.sh` prints "could not detect host IP" | No `eth0` and `hostname -I` returned nothing | Pass explicitly: `HOST_IP=192.168.x.y ./up.sh` |
| Running `docker compose up -d` directly errors on prometheus.yml | The file was regenerated with a stale IP or doesn't exist | Always use `./up.sh` — it regenerates the file every run |
| Grafana shows "No data" on all panels | Datasource not connecting | Datasources → Prometheus → **Test**. If red, edit URL to `http://kafka-lab-prometheus:9090` |
| Dashboard missing from Grafana UI | Provider didn't load | `docker compose logs grafana \| grep -i dashboard`; check volume mount paths are readable |
| Metric names in dashboard don't match `/metrics` output | JMX config file changed after provisioning | Restart Prometheus (`docker compose restart prometheus`) and Grafana (`docker compose restart grafana`) |
| Port 3000 or 9090 in use | Another Grafana / Prometheus (or dev app) is running | `ss -ltnp \| grep -E ':(3000\|9090)\b'`; stop the other one, or edit `docker-compose.yml` port mappings |
| Broker fails to start after `start-monitoring` | Wrong path in `JMX_EXPORTER_JAR` / `_CONFIG` | Check `logs/broker-N.log` — a `FileNotFoundException` on the jar or config tells you which |
| High CPU on broker after enabling monitoring | JMX config too broad (matching thousands of MBeans) | This config is scoped — but if you swap in the upstream `kafka-2_0_0.yml`, expect noticeable overhead |

## 12. Shutting down cleanly

```bash
# Prometheus + Grafana
cd Lab-02-Monitoring-Setup
./down.sh

# Kafka brokers (unchanged from Lab-01)
cd ../kafka-lab
./cluster.sh stop
```

To wipe Prometheus/Grafana state and start fresh:

```bash
cd Lab-02-Monitoring-Setup
./down.sh -v
```

To disable monitoring on the next start-up: just use `./cluster.sh start` (or `start-broker N`) instead of the `-monitoring` variants. Broker JVMs come up without the JMX agent attached; nothing else changes.

**After rebooting WSL** the host IP often changes. Re-run `./up.sh` to regenerate `prometheus.yml` with the new IP.
