Section 3

Topic Anatomy

Concept → Partitions → Records → Ordering
3.1 Concept

A topic is a named, append-only stream

orders.events — one name, one purpose, one owner

Applications use the name. Kafka stores the data inside partitions.

3.2 Partitions

A topic is divided into partitions

P0
0 · 1 · 2 · 3 · …
P1
0 · 1 · 2 · 3 · …
P2
0 · 1 · 2 · 3 · …
  • Independent ordered logs
  • Distributed across brokers
  • Parallel reads and writes
The partition — not the topic — is Kafka's unit of ordering & parallelism.
3.3 Offsets

An offset is a position inside one partition

orders.events-2 @ 1057
  • Starts at 0, monotonically increases
  • Unique within a partition — not across the topic
  • Consumers track progress by offset
  • Never reset when old segments are deleted
3.4 Records

What a record carries

Payload

  • Key
  • Value
  • Timestamp
  • Headers

Positioning

  • Topic name
  • Partition
  • Offset

Everything except the payload is metadata Kafka assigns.

3.5 Ordering

Ordering guarantee

✓ Guaranteed

Within one partition

✕ Not guaranteed

Across partitions of the same topic

Design rule Use orderId as the record key → all events for one order share a partition → stay ordered.
3.6 Partitioner

How a producer picks a partition

  1. Explicit partition provided → use it
  2. Key provided → hash it
  3. No key → sticky / round-robin for throughput
partition = hash(key) mod partitionCount
Admin implication Partition count is baked into application behaviour. Changing it affects future key placement.
3.7 Hot partition

The hot partition problem

P0
1k msg/s
P1
50k msg/s ⚠
P2
1k msg/s

Common causes

  • Low-cardinality key (country, tenantId)
  • One customer producing 80% of the traffic
  • Constant key set by mistake
  • Time-based key concentrating current traffic
Trap Adding partitions does not fix a bad key distribution.
Section 4

Choosing Partition Count

The one decision you can't cleanly undo
4.1 What partitions control

Five dimensions of the decision

1Throughput 2Parallelism 3Ordering 4Operations 5Growth
No universal "correct number" — balance all five.
4.2 Inputs

Measure first, calculate second

InputExample
Peak records/s60,000
Avg serialized record size2 KB
Peak ingress120 MB/s
Required consumer groups4
Slowest group / instance10 MB/s
Ordering keyorderId
Annual growth40%
Replication factor3

Use peak, not average. Measure serialized/compressed bytes.

4.3 Formula · write side

How many partitions to accept the writes?

Pwrite = Peak ingress Safe write MB/s per partition

Use a sustainable rate — not the highest burst.

4.4 Example · write side

Applied to orders.events

Given Peak ingress = 120 MB/s
Safe write per partition = 15 MB/s
Pwrite = 120 15 = 8 partitions
Safe means Target latency met · replication stable · no request-queue growth · no ISR flap.
4.5 Formula · read side

How many partitions to feed the consumers?

Pread = Group throughput Safe MB/s per consumer instance

Per consumer group. Take the highest-demand group.

4.6 Example · read side

Fraud-detection group

Given Group throughput = 120 MB/s
Safe per-consumer = 10 MB/s
Pread = 120 10 = 12 partitions
4.7 The rule

Take the maximum

Partitions = max(Pwrite, Pread, required concurrency)
Our example max(8, 12) = 12 partitions (initial)

Then adjust for skew, growth & broker capacity ↓

4.8 Consumer parallelism

Consumers per group ≤ partitions

Partition ↔ consumer within one group is 1:1 at any moment.

PartitionsConsumers in groupResult
1243 partitions / consumer
12121 partition / consumer
121612 active · 4 idle
Note Extra consumer groups are independent — each gets its own read of every partition.
4.9 Read load × groups

Multiple groups multiply the read load

Scenario orders.events ingress = 120 MB/s, four full-volume groups (fraud, inventory, notifications, analytics)
Total reads = 120 × 4 = 480 MB/s

Partition count must fit broker network & disk-read capacity — not only ingress.

4.10 Compression

Compression changes what "MB/s" means

Raw 200 MB/s, 2.5:1 codec ≈ 80 MB/s on the wire

Effectiveness depends on

  • Record similarity
  • Batch size
  • Codec (gzip / snappy / lz4 / zstd)
  • Producer linger.ms
  • Already-compressed payloads
  • CPU availability
Test with realistic data — don't size from a compression brochure.
4.11 Replication multiplier

Replication multiplies internal traffic

Cluster write bytes ≈ Ingress × RF
120 MB/s ingress · RF 3 ≈ 360 MB/s of data movement

Add protocol overhead, acks, cross-rack traffic on top.

4.12 Broker distribution

Does the count fit the cluster?

12 partitions × RF 3 across 6 brokers Total replicas = 12 × 3 = 36
Replicas / broker = 36 ÷ 6 = 6
Leaders / broker = 12 ÷ 6 = 2
Sanity, not balance Also check byte size, traffic, rack placement, and topics already on each broker.
4.13 Storage sizing

How much disk will this cost?

Primary retained ≈ ingress × retention seconds
Replicated ≈ primary × RF
20 MB/s · 7 days · RF 3 Primary ≈ 12.1 TB≈ 36.3 TB replicated
4.14 Per-partition size

Average partition size matters

Size / partition = Primary retained Partition count
Primary dataPartitionsAvg / partition
12 TB62 TB — slow recovery
12 TB121 TB — reasonable
12 TB24500 GB — many files
4.15 Recovery time

Recovery gate on partition size

Recovery time = Partition size Safe transfer rate
500 GB · 50 MB/s ≈ 2.8 hours
Real recovery is slower: production traffic + multiple replicas share the same disks & NIC.
4.16 Growth headroom

Don't size only for today

Final = Pcalc × (1 + growth)
12 partitions · 40% growth 12 × 1.4 = 16.8Round to 18 partitions
Avoid over-partitioning "just in case" — every partition has a cost.
4.17 The reordering trap

Why you can't just double later

hash(key) mod 6   ≠   hash(key) mod 12

Before

ORD-1000 → P3
ORD-1000 → P3
ORD-1000 → P3

After growth

ORD-1000 → P3 (historical)
ORD-1000 → P9 (future)
Two consumers, undefined order.

Cost Per-key ordering is broken across the boundary. Stateful consumers rebuild wrong state.
4.18 Key cardinality

Partitions can't fix bad keys

Symptom 24 partitions · key = country · 5 country values dominate → 5 hot partitions, 19 idle.

Better keys

  • Higher cardinality (customerId, sessionId)
  • Composite (country + customerId)
  • Bucketed (tenantId + bucket)
4.19 Skew safety

Design for the hottest partition

Hottest = avg × skew factor
Reality check Avg planned = 10 MB/s · Observed skew = 1.8×
Hottest partition = 18 MB/s — safe capacity = 15 MB/s Unsafe.
Total topic throughput can look fine while one partition is on fire.
4.20 Add or split?

Add partitions or split into a new topic?

Add partitions when

  • Consumer parallelism is the bottleneck
  • Per-partition throughput is near safe cap
  • Key distribution is healthy
  • Brokers have capacity

Split into a new topic when

  • Retention differs
  • Security / classification differs
  • Ordering scope differs
  • Owners / lifecycles differ

orders.events (7-day history) vs orders.state (compacted latest state)

4.21 Decision flow

The 10-step decision

  1. Measure peak byte + record rate
  2. Benchmark safe write / partition
  3. Calculate Pwrite
  4. Benchmark slowest consumer
  5. Calculate Pread
  6. Take the larger
  7. Validate key cardinality & skew
  8. Add growth headroom
  9. Check brokers, storage, recovery
  10. Review ordering impact · approve
4.22 Cost of over-partitioning

More partitions are not free

  • Controller metadata workload
  • Open files & segment count
  • Memory usage
  • Leader-election work
  • Replication connections
  • Recovery time after broker loss
Follow the workload — not a rule of thumb like "always 50".
Section 5

Replication & Durability

Leader · Followers · ISR · min.ISR · acks
5.1 Replication factor

Replication factor = number of copies

3 partitions · RF 3 → 9 replicas total

Broker 1
P0-L P1 P2
Broker 2
P0 P1-L P2
Broker 3
P0 P1 P2-L

One replica per partition is the leader; others are followers.

5.2 ISR

In-Sync Replicas — who's caught up

Healthy

Replicas: 1,2,3
ISR: 1,2,3

Degraded

Replicas: 1,2,3
ISR: 1,2

Why ISR shrinks

  • Broker failure · network latency
  • Slow / failed disk · GC pause
  • Broker overload · throttling
5.3 Durability triangle

Three knobs, one guarantee

RF

= 3

Copies exist

min.ISR

= 2

Must be in-sync

acks

= all

Wait for ISR

Any one weakened → the guarantee collapses.
min.ISR matters only when producer uses acks=all.
5.4 The safety formula

Failures you can tolerate

RF ≥ min.ISR + F

where F = concurrent broker failures without producer errors

RFmin.ISRF toleratedTrade-off
312 downAvailability > durability (weak)
321 downBalanced (recommended)
330 downBrittle — first ISR flap fails writes
5.5 Broker failure

What happens when broker 1 dies

Broker 1 ✕
P0-L
Broker 2
P0-L (new)
Broker 3
P0
  1. Controller detects failure
  2. Elects a follower from the ISR
  3. Clients refresh metadata
  4. Failed replica stays out of ISR until it catches up
Recovery quality = replication health before the failure.
5.6 Unclean election

Availability vs durability trade-off

What if no in-sync replica is available?

unclean=false

Partition stays offline
Wait for an ISR replica
Zero data loss

unclean=true

Elect an out-of-sync replica
Partition serves again
Silent data loss

Define policy by workload criticality. Don't enable casually cluster-wide.
Section 6

Topic Operations

Naming · Create · Describe · Configure
6.1 Naming

Use a predictable convention

<domain>.<entity>.<event-type>.<v>
  • commerce.orders.events.v1
  • payments.transactions.events.v1
  • inventory.stock.compacted.v1
Standards for: characters/case · env separation · ownership · classification · versioning · DLQ · reserved prefixes.
6.2 Pre-create checklist

Decide BEFORE you type --create

  • Business + technical owner
  • Peak throughput & record size
  • Partition key + ordering scope
  • Partition count (with math)
  • Replication factor + rack placement
  • Retention / compaction
  • Max record size
  • ACLs
  • Monitoring + cost + deletion plan
6.3 Create

Every production create is explicit

bin/kafka-topics.sh --bootstrap-server kafka-1:9092 \
  --create --topic orders.events \
  --partitions 12 \
  --replication-factor 3 \
  --config min.insync.replicas=2 \
  --config cleanup.policy=delete \
  --config retention.ms=604800000
Never leave defaults implicit. Capture in version control.
6.4 Describe

Read the output

Topic: orders.events  Partition: 0  Leader: 2
Replicas: 2,3,1       Isr: 2,3,1

Healthy

ISR == Replicas
Leader present

Warning

Leader: -1
ISR < Replicas
One broker leading too many partitions

6.5 Filters

Only show problems

--describe --under-replicated-partitions
--describe --unavailable-partitions
--describe --under-min-isr-partitions
CLI is for investigation. Use monitoring (Prometheus + Grafana) for continuous detection.
6.6 Config precedence

Where does an effective value come from?

PrioritySourceHow to set
1 (highest)Dynamic topic overridekafka-configs --entity-type topics --alter
2Dynamic broker (per / default)--entity-type brokers --entity-default
3Static brokerbroker-N.properties + restart
4 (lowest)Kafka code defaultBaked in
Use --describe --all to see the source (synonyms=) — not just the value.
6.7 Modify

Change & revert

kafka-configs.sh --alter --entity-type topics \
  --entity-name orders.events \
  --add-config retention.ms=1209600000
kafka-configs.sh --alter --entity-type topics \
  --entity-name orders.events \
  --delete-config retention.ms
Always record the previous value and the rollback command.
Section 7

Retention, Compaction
& Storage

7.1 Two policies

Compare

delete

  • Removes old segments
  • By time (retention.ms)
  • Or size (retention.bytes)
  • Event history use case

compact

  • Keeps latest value per key
  • Async cleaner
  • Requires keys
  • State / changelog use case

compact,delete

  • Compact + age-out
  • Bounded changelog
7.2 Time retention

Retention operates on segments

retention.ms = 604 800 000
(7 days)
Trap Retention deletes whole rolled segments. The active segment is never touched.
→ data can outlive retention.ms by segment.ms.
7.3 Size retention

Per-partition, not per-topic

Topic limit ≈ retention.bytes × P × RF
Watch A high-volume burst can shorten the effective time window.
7.4 Segments

The unit that gets deleted or compacted

Seg 0
rolled
Seg 632
rolled
Seg 1240
ACTIVE
  • Rolls on segment.bytes or segment.ms
  • Retention deletes whole rolled segments
  • Compaction rewrites whole rolled segments
  • The active segment is immune
7.5 Compaction

Latest value per key wins

Before

C101 → Silver
C102 → Gold
C101 → Gold

After

C102 → Gold
C101 → Gold
  • Async — not immediate
  • Keys are mandatory
  • Offsets are preserved
7.6 Tombstones

Delete a key with a null value

C101 → null
  • Signals "logically delete this key"
  • Cleaner removes older records for that key
  • Tombstone lives for delete.retention.ms (default 24h)
Consumers offline longer than the tombstone window rebuild wrong state.
7.7 Message size handshake

Five layers must agree

LayerSettingDefault
Producermax.request.size1 MB
Brokermessage.max.bytes1 MB
Topicmax.message.bytes= broker
Replicationreplica.fetch.max.bytes1 MB
Consumermax.partition.fetch.bytes1 MB
Strictest wins. Change one, forget the others → cascading errors.
Section 8

Partition Growth
& Reassignment

8.1 Add partitions

Command is easy — the impact is not

kafka-topics.sh --alter \
  --topic orders.events \
  --partitions 18
  • Existing data is not redistributed
  • New partitions start empty
  • Future keys may map to different partitions
  • Consumer groups rebalance
Application-visible change. Coordinate with the app team.
8.2 Can't shrink

One-way ratchet

Error: Number of partitions cannot be reduced.
If you must shrink 1. Create a new topic with fewer partitions
2. MirrorMaker 2 the data across
3. Cut consumers over
4. Delete the old topic
→ weeks-long migration
8.3 Reassignment

Used for

  • Move replicas between brokers
  • Rebalance after adding brokers
  • Drain a broker before decommission
  • Move replicas between log dirs
  • Change the replication factor

Phases

  1. Capture current assignment
  2. Build & review plan
  3. Execute (throttled)
  4. Monitor
  5. Verify & remove throttles
8.4 Throttling

Don't self-DoS

--throttle 50000000 (bytes / sec)

Too high

Client latency spikes
Brokers saturate

Too low

Reassignment takes forever
Cluster in half-state

Cleanup Always --verify. It removes throttle configs when done. Skip it → throttles persist forever.
Section 9

Delete, Backup
& Governance

9.1 Delete safeguards

Deletion is destructive — treat accordingly

  • Owner approval required
  • Confirm exact cluster & topic name
  • Verify dependencies (consumers, dashboards, alerts)
  • Export required data / metadata first
  • Least-privilege ACLs — no ad-hoc admin
  • Automation with review gates
Never use broad shell patterns (kafka-topics --delete --topic 'test.*') in prod.
9.2 Not a backup

Replication ≠ backup

Replication does not protect against:

  • Accidental topic deletion
  • Wrong retention change
  • Corrupt data from a bad producer
  • Cluster-wide disaster
  • Operator error replicated everywhere
Real DR options Cross-cluster replication (MM2, Cluster Linking) · immutable downstream storage · tested runbooks
9.3 Internal topics

Kafka's own topics — protect them

  • __consumer_offsets
  • __transaction_state
  • Kafka Connect config / offsets / status
  • Kafka Streams changelog / repartition
Deleting these breaks the platform even when business topics look healthy.
Monitor their availability & ISR like production topics.
9.4 Auto-create trap

The silent shadow-topic problem

Scenario App configured to write to orders.events but a typo sends to order.events.
With auto.create.topics.enable=true, Kafka creates it silently: 1 partition · RF 1 · default retention

Production posture

  • auto.create.topics.enable=false
  • Enforce via ACLs on Create
  • Create only via automation

Detection

  • Weekly topic audit
  • Alert on new topics
  • Compare vs approved catalog
Section 10

Monitoring &
Troubleshooting

10.1 Metrics that matter

The topic-level signals

Availability

  • Offline partitions
  • Active controllers
  • Under-min-ISR count

Health

  • URP count
  • ISR shrink / expand rate
  • Leader & replica skew

Throughput

  • Bytes in / out per topic
  • Request latency
  • Consumer lag
10.2 Health states

Four states to distinguish

StateSignalMeaning
✓ HealthyReplicas 1,2,3 · ISR 1,2,3Full redundancy
⚠ DegradedReplicas 1,2,3 · ISR 1,2Redundancy reduced
⚠ At min.ISRISR = min.ISROne more loss → writes fail
✕ OfflineLeader = -1Partition unavailable
Don't wait for offline. Under-replicated is the early warning.
10.3 URP triage

Under-replicated partition — investigate

  1. Which topics · partitions · brokers?
  2. Is a broker offline / restarting?
  3. Disk capacity, errors, latency?
  4. Network — connectivity, saturation?
  5. CPU, memory, JVM pauses?
  6. Replication lag & fetcher behavior?
  7. Recent reassignments or config changes?
  8. Does replica rejoin ISR after recovery?
Avoid forcing leader elections before understanding replica freshness.
10.4 Skew is not one thing

Diagnose the type of skew first

Data skew

  • Record count
  • Byte size
  • Traffic

Placement skew

  • Leader distribution
  • Replica count / broker

Consumer skew

  • Consumer lag concentrated on one partition
Different skew → different fix. Don't reach for reassignment when the answer is a better key.
10.5 Common mistakes

The greatest hits

  • Creating topics with broker defaults, no review
  • Believing RF alone = durability
  • Increasing partitions without checking key ordering
  • Treating ISR shrink as harmless
  • Unlimited retention with no capacity plan
  • Compaction without meaningful keys
  • Forgetting to remove reassignment throttles
  • Deleting topics without checking consumers
  • Changing many risk-sensitive settings at once
  • Managing prod topics via ad-hoc CLI
Section 11

Governance
& Wrap-Up

11.1 Design review

Approve a topic only after…

  • Clear name, purpose, owner
  • Data classification & access policy
  • Partition key & ordering scope
  • Peak throughput & record size
  • Partition count with math
  • RF + rack placement
  • min.ISR + producer acks
  • Cleanup / retention / compaction
  • Capacity & cost estimate
  • Monitoring + alerting + lifecycle
11.2 Change runbook

Every topic change follows the same 10 steps

  1. Objective & success criteria
  2. Capture current state
  3. Impact & capacity check
  4. Exact commands + rollback
  5. Peer / owner approval
  6. Execute in approved window
  7. Monitor topic + brokers + clients
  8. Verify config & health
  9. Update infra-as-code + docs
  10. Close after post-change observation
11.3 Key takeaways

What to remember

  • A topic is distributed partition logs
  • Ordering: within partition, never across
  • Partition count scales but costs — size with math
  • RF · min.ISR · acks work together
  • Retention ≠ compaction — pick per workload
  • Partitions grow but never shrink
  • Every change is application-visible → automate + govern
  • Healthy topics = healthy platform
Design once, with math.
Monitor always.
Change with a runbook.
Never with vibes.
