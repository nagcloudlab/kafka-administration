Commands you'll reuse

Inspect & modify — the two commands

Inspect a topic's effective config (with source)

kafka-configs.sh --bootstrap-server $BS \
  --entity-type topics --entity-name orders.events \
  --describe --all

Set an override

kafka-configs.sh --bootstrap-server $BS \
  --entity-type topics --entity-name orders.events \
  --alter --add-config retention.ms=1209600000

Remove an override (revert to broker default)

kafka-configs.sh --bootstrap-server $BS \
  --entity-type topics --entity-name orders.events \
  --alter --delete-config retention.ms
Chapter 1

Retention & Compaction

How data leaves your topic
7 properties
1.1 Cleanup policy

cleanup.policy

default: delete dynamic

What it does

  • Selects how old data is removed: delete, compact, or delete,compact
  • Determines whether the log cleaner runs on this topic

When to change

Event history / audit → delete. Current-state / changelog → compact. Bounded state with age cap → delete,compact.
Trap compact requires keys on every record. Null-key records are ignored by the cleaner and live forever. 
--config cleanup.policy=compact
1.2 Time retention

retention.ms

default: 604800000 (7 d) dynamic

What it does

  • How long a rolled segment can exist before deletion (delete-policy only)
  • Measured from the segment's largest message timestamp
  • -1 = keep forever

When to change

Longer for replay windows / compliance · shorter to bound disk cost.
Trade-off Retention only deletes rolled segments — the active segment is immune. Data can outlive retention.ms by up to segment.ms. 
--config retention.ms=1209600000    # 14 days
1.3 Size retention

retention.bytes

default: -1 (unlimited) dynamic

What it does

  • Max bytes to retain per partition (not per topic!)
  • Oldest segments removed once total exceeds this
Per partition Actual topic disk ≈ retention.bytes × partitions × RF. A 6-partition topic with RF=3 and retention.bytes=1GB can use ~18 GB.
Trade-off High-volume bursts shorten the effective time window. Prefer time OR size — not both — for reasoning. 
--config retention.bytes=10737418240    # 10 GB per partition
1.4 Tombstone retention

delete.retention.ms

default: 86400000 (24 h) dynamic

What it does

  • How long tombstones (key + null-value) remain visible in compacted topics
  • Gives downstream consumers time to see the deletion before the cleaner removes it
Only relevant when cleanup.policy=compact.
Trade-off Consumers offline longer than this window never see the delete → rebuild wrong state. 
--config delete.retention.ms=604800000    # 7 days (safer)
1.5 Compaction eligibility

min.cleanable.dirty.ratio

default: 0.5 dynamic

What it does

  • Cleaner ignores compacted partitions until dirty portion / total ≥ this ratio
  • Prevents the cleaner from running on every tiny change

When to change

Lower (e.g. 0.1) → more aggressive cleaning, less duplicate history, more I/O.
Higher (e.g. 0.75) → cleaner runs less often, larger disk footprint.
Test Values below 0.01 cause continuous cleaner activity — often thrashing. 
--config min.cleanable.dirty.ratio=0.1
1.6 Min compaction lag

min.compaction.lag.ms

default: 0 dynamic

What it does

  • Records younger than this are protected from compaction
  • Guarantees a minimum "recent history" window for consumers

When to change

Set when consumers need to see every intermediate value for a key for at least N seconds/minutes (e.g. CDC replay, debugging window). 
--config min.compaction.lag.ms=3600000    # 1 h protected
1.7 Max compaction lag

max.compaction.lag.ms

default: Long.MAX_VALUE dynamic

What it does

  • Forces compaction after this age even if dirty ratio hasn't been reached
  • Bounds the maximum "how stale can a duplicate be" delay

When to change

Set when compliance/GDPR requires "deleted keys must actually be gone within N days".
Interaction Overrides min.cleanable.dirty.ratio when triggered. Increases cleaner load. 
--config max.compaction.lag.ms=2592000000    # 30 days
Chapter 2

Segments

The physical unit of retention & compaction
4 properties
2.1 Segment size

segment.bytes

default: 1073741824 (1 GB) dynamic

What it does

  • Rolls the active segment when it reaches this size
  • Determines granularity of retention deletion & compaction

When to change

Smaller (64–256 MB) → faster response to retention, more compaction opportunities, but more open files.
Larger (1–4 GB) → default for busy topics — fewer segments, less overhead.
Trade-off Never below 16 MB in production. Small segments multiply open file handles & index memory. 
--config segment.bytes=268435456    # 256 MB
2.2 Segment age

segment.ms & segment.jitter.ms

segment.ms default: 604800000 (7 d) jitter default: 0 dynamic

What it does

  • segment.ms — rolls the segment after this age, even if size not reached
  • segment.jitter.ms — random offset up to this value, to prevent every partition rolling at the same instant

When to change

Low-volume topics may never hit segment.bytes — set segment.ms=86400000 (1 day) so retention can eventually delete.
Ops tip Add jitter (~10% of segment.ms) to prevent a "roll storm" on large clusters where thousands of partitions all roll simultaneously. 
--config segment.ms=86400000
--config segment.jitter.ms=3600000    # ±1h jitter
2.3 Index size

segment.index.bytes

default: 10485760 (10 MB) dynamic

What it does

  • Max size of the offset index file per segment (pre-allocated)
  • Larger index = more index entries = faster seeks in big segments

When to change

Rarely tuned. Reduce (e.g. 1 MB) for clusters with many thousands of small partitions to save mmap memory.
Interaction Works with index.interval.bytes (broker-level, default 4 KB) which controls how sparse the index is. 
--config segment.index.bytes=1048576    # 1 MB
2.4 File deletion delay

file.delete.delay.ms

default: 60000 (60 s) dynamic

What it does

  • Time to wait after marking a segment file for deletion before actually removing it
  • Gives in-flight reads time to finish

When to change

Almost never. Increase if you see consumer errors like Log file was deleted during very slow reads. 
--config file.delete.delay.ms=120000    # 2 min
Chapter 3

Messages & Size

What the topic will accept on the wire
3 properties
3.1 Max message size

max.message.bytes

default: 1048588 (~1 MB) dynamic

What it does

  • Largest record batch the broker will accept for this topic
  • Overrides broker-level message.max.bytes for this topic
Five-layer handshake Producer max.request.size, broker message.max.bytes, topic max.message.bytes, broker replica.fetch.max.bytes, consumer max.partition.fetch.bytes — strictest wins.
Rule Never let this exceed the broker's replica.fetch.max.bytes. Replication will silently stall. 
--config max.message.bytes=4194304    # 4 MB
3.2 Compression

compression.type

default: producer dynamic

What it does

  • producer — preserve whatever codec the producer chose (default, most flexible)
  • none, gzip, snappy, lz4, zstd — force this codec, recompressing if producer used a different one
  • uncompressed — force plain, even if producer sent compressed
Trade-off Forcing recompression consumes broker CPU. Prefer producer unless you have a specific need to standardise (e.g. compliance, cost). 
--config compression.type=zstd    # smaller on disk, more CPU
3.3 Down-conversion

message.downconversion.enable

default: true dynamic

What it does

  • Broker down-converts message format for old consumers that don't support the current format
  • Very old clients (pre-0.11) may still request the old format

When to change

Set to false once all clients are modern (Kafka 2.0+). Saves broker CPU & memory when old fetches would trigger conversion.
Verify first Old clients will get UNSUPPORTED_VERSION errors. Confirm zero legacy clients before disabling. 
--config message.downconversion.enable=false
Chapter 4

Timestamps

Retention math depends on these
2 properties
4.1 Timestamp source

message.timestamp.type

default: CreateTime dynamic

What it does

  • CreateTime — use the timestamp the producer supplied (event time)
  • LogAppendTime — broker overwrites with wall-clock at append time

When to change

Use LogAppendTime when producers can't be trusted to send monotonically-increasing timestamps (embedded devices, backfill jobs).
Impact retention.ms uses the segment's largest timestamp. If you switch to LogAppendTime, retention becomes "seconds since data landed" not "seconds since event". 
--config message.timestamp.type=LogAppendTime
4.2 Timestamp bounds

message.timestamp.before.max.ms & after.max.ms

default: Long.MAX_VALUE dynamic Kafka 3.0+

Replaces the older message.timestamp.difference.max.ms single-value setting.

What they do

  • Reject records whose timestamp is more than N ms in the past (before.max.ms) or future (after.max.ms) relative to broker wall-clock
  • Only enforced when message.timestamp.type=CreateTime
Protects retention math from producers with wildly wrong clocks. A rogue timestamp 100 years in the future would make everything after it un-deletable. 
--config message.timestamp.before.max.ms=3600000    # reject > 1h old
--config message.timestamp.after.max.ms=300000      # reject > 5min future
Chapter 5

Durability

The write-side safety net
2 properties
5.1 Min in-sync replicas

min.insync.replicas

default: 1 dynamic

What it does

  • Minimum ISR count required to accept an acks=all write
  • If ISR drops below this, producer gets NotEnoughReplicasException
Only with acks=all Ignored when producer uses acks=0 or acks=1. Setting min.ISR alone doesn't protect writes.
Safety formula RF ≥ min.ISR + F → tolerates F broker failures without producer errors.
Recommended: RF=3 · min.ISR=2 · acks=all. 
--config min.insync.replicas=2
5.2 Unclean leader election

unclean.leader.election.enable

default: false (since 0.11) dynamic

What it does

  • When ISR is empty, can Kafka elect an out-of-sync replica as leader?
  • false → partition stays offline until an ISR member returns (durability > availability)
  • true → elect any surviving replica (availability > durability, with silent data loss)
Never casually Only enable per-topic, for workloads where availability strictly outranks a possible data-loss event (metrics, logs, cache warm-up streams). 
--config unclean.leader.election.enable=true    # metrics-tier only
Chapter 6

Replication Throttling

Live during a reassignment or planned catch-up
2 paired properties
6.1 Throttled replica lists

follower / leader .replication.throttled.replicas

default: (empty) dynamic

What they do

  • Marks specific partition:broker replicas as throttled during replication
  • Actual rate cap set by broker-level follower.replication.throttled.rate / leader.replication.throttled.rate
  • Set automatically by kafka-reassign-partitions --execute --throttle N
Cleanup matters Skipping kafka-reassign-partitions --verify means these configs stay set forever — chronically slow followers, mysterious URP. 
# format: "partitionId:brokerId,partitionId:brokerId,..." or "*"
--add-config follower.replication.throttled.replicas=0:102,1:103
Chapter 7

Flush & Preallocation

Rarely tuned — know they exist
3 properties
7.1 Flush controls

flush.messages · flush.ms · preallocate

flush.messages: Long.MAX_VALUE flush.ms: Long.MAX_VALUE preallocate: false dynamic

What they do

  • flush.messages — force fsync after every N messages
  • flush.ms — force fsync every N ms
  • preallocate — allocate segment file size up-front (Windows only, rarely needed)
Trap: don't tune these Kafka's durability is designed around replication, not fsync. Setting flush.messages=1 for "safety" kills throughput and is redundant with acks=all + min.ISR.
Use replication + acks=all for durability. Leave flush at defaults on Linux.
Chapter 8

Tiered Storage (KIP-405)

Kafka 3.6+ · Early Access
3 properties
8.1 Enable tiering

remote.storage.enable

default: false dynamic Kafka 3.6+

What it does

  • Enables offload of rolled segments to remote storage (S3, GCS, Azure Blob, ...)
  • Requires remote.log.storage.system.enable=true at broker level + a plugin
  • Compacted topics can't be tiered (yet)
One-way Cannot be disabled once data is on remote — you'd need to migrate to a new topic. 
--config remote.storage.enable=true
8.2 Local hot tier

local.retention.ms · local.retention.bytes

default: -2 (same as retention) dynamic Kafka 3.6+

What they do

  • How much data to keep on local broker disk before offloading
  • retention.ms / retention.bytes still control total (local + remote)
  • Only meaningful when remote.storage.enable=true
Set local.retention.ms to your consumer's typical tail-lag SLO. Real-time consumers get fast local reads; archive replays hit slower remote fetch. 
--config local.retention.ms=3600000     # 1 h hot
--config retention.ms=2592000000        # 30 d total

The 5 you shouldn't touch

Leave-alone list

  • flush.messages · flush.ms — replication IS your durability, not fsync
  • preallocate — Windows-only workaround, irrelevant on Linux
  • segment.index.bytes — 10 MB default is fine unless you have > 10k partitions
  • file.delete.delay.ms — 60 s default fine unless very slow consumers
  • message.downconversion.enable — leave true unless you've verified zero legacy clients
Rule If a property doesn't have a documented workload trigger to change it, don't.
