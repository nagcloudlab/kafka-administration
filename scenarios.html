Lab 03Broker Lifecycle & Availability

Three failure modes an Ops team pages on: graceful vs ungraceful shutdown, the stale-broker-epoch trap, and reading the URP alert without over-reacting.
3.1

Controlled shutdown vs kill -9

ops

Problem: Runbook says "kill -9 if slow." Wrong. SIGTERM migrates leaderships before releasing the socket → clients see one clean redirect (<100 ms). SIGKILL skips migration → clients hammer a dead leader until ZK session expires (5–15 s of errors).

Demo: 4 terminals; producer at 2000 rec/s acks=all; graceful stop → records/sec stays flat · kill -9 → records/sec drops to zero for ~10 s.

SIGTERM migrates leadership. SIGKILL migrates client errors.
Open Scenario 3.1
3.2

Stale broker epoch (the incident, on purpose)

ops

Problem: Cluster is running. Operator runs ./cluster.sh start again → NodeExistsException on /brokers/ids/N — the live znode. "Cleans up" the znode → live brokers become fenced zombies. Producer hangs; consumer sees nothing.

Demo: reproduce today's real incident. Diagnosis: ps → ss -tlnp → znode timestamp. Recovery: kill -9 (the one Chapter 3 case where SIGKILL is correct).

Ephemeral znodes belong to live sessions, not dead processes. Check the port before you delete.
Open Scenario 3.2
3.3

Under-replicated partitions (URP is not a page)

ops

Problem: URP fires with two very different root causes — broker gone (real risk) vs slow follower (no risk). Same metric, same alert. Operators either wake up at 3 AM for a lagging follower, or (worse) stop responding entirely.

Demo: throttle broker 102's follower fetches to 1 KB/s using follower.replication.throttled.rate. ISR shrinks; URP fires. Triage: ss + log-dirs + configs.

URP = "behind on data", not "data at risk". The real page metric is UnderMinIsrPartitionCount. Triage before you page.

Lab 04Replication & Durability

The write-path safety net: min.insync.replicas, acks=all, unclean leader election, rack awareness. The three durability decisions every topic makes.
4.1

min.insync.replicas + acks=all

durability

Problem: Three knobs (RF, min.ISR, acks) interact to define durability. "Higher = safer" is wrong — min.ISR=3 on RF=3 is brittle, not durable.

Demo: three topics (safe / brittle / risky) with different min.ISR (2 / 3 / 1). Stop one broker; watch the three producers react differently — one keeps working, one fails, one keeps working but silently at risk.

RF − min.ISR = failures you tolerate. Match to the workload's durability need.
Open Scenario 4.1
4.2

Unclean leader election (silent data loss)

durability

Problem: If ISR goes empty, does Kafka elect an out-of-sync replica (availability with data loss) or wait (durability with downtime)?

Demo: twin topics (unclean=true / false), same failure sequence. Watch the offsets probe: unclean topic silently drops from :100 to :50; clean topic stays offline until the leader returns.

Unclean election trades durability for availability. Silent by default. Alert on every occurrence.
Open Scenario 4.2
4.3

Rack awareness (placing replicas across failure domains)

durability

Problem: Default placer is rack-blind. All 3 replicas can end up in the same rack → single rack failure = data loss. Adding broker.rack later doesn't retroactively fix existing topics.

Demo: add broker.rack=r1/r2/r3, rolling restart. Create a new topic → rack-aware placement. Compare with a pre-rack topic (unchanged). Fix via kafka-reassign-partitions.

Set broker.rack cluster-wide at stand-up. Otherwise you're one accidental rack fault from an unclean election.
Open Scenario 4.3

Lab 05Data Lifecycle

How data leaves your topic (retention vs compaction) and how to move it between brokers without hurting production traffic.
5.1

Retention vs compaction (what stays, when)

storage

Problem: Two cleanup policies with different mental models. Common traps: retention deletes SEGMENTS (not messages); compaction requires KEYS (null-keyed records live forever); cleaner-death is silent.

Demo: two topics (delete vs compact) with tiny segments (segment.ms=10s). Produce keyed records; wait; consume both — delete topic drops old segments, compact topic keeps latest-per-key only.

Delete deletes segments (not messages). Compact keeps latest per key (requires keys). Different tools — pick per workload.
Open Scenario 5.1
5.2

Partition reassignment without saturating the network

storage

Problem: Moving partitions between brokers can spike replication traffic and starve clients. Skipping --verify leaves throttle configs applied forever.

Demo: two topics filled with 50 MB each; reassign one with --throttle 5MB/s, the other unthrottled. Time both via date. Watch broker BytesIn/s via JMX. Clean up throttles with --verify.

Reassignment without --throttle is a self-inflicted DoS. Without --verify, the throttle never leaves.
Open Scenario 5.2

Lab 06Client Isolation

Consumer-side rebalance storms and producer/consumer quotas — how to keep one client from ruining the party for everyone else.
6.1

Consumer group rebalance storm

clients

Problem: Short session.timeout.ms + real GC pause = endless rebalance. Consumer keeps getting evicted → rejoins → evicts others. Group processes nothing.

Demo: 3 consumers with session.timeout.ms=6000; SIGSTOP one for 10 s (fake GC pause), then SIGCONT. Watch two rebalances happen per pause. Contrast RangeAssignor vs CooperativeStickyAssignor.

Rebalance storms are a client-tuning problem. Short session.timeout.ms + real GC = storm. Use cooperative + static membership.
Open Scenario 6.1
6.2

Quotas (protecting a shared cluster from one runaway client)

clients

Problem: One misbehaving client can saturate broker NIC or CPU. Client-id quotas are honour-only (anyone can pick any client-id); user quotas require SASL; default quotas are the real safety net.

Demo: baseline perf-test (50 MB/s) → apply producer_byte_rate=1MB/s to the client-id → rerun (~1 MB/s). Prove same test with a different client-id runs full speed → set default quota → all clients capped.

Quotas are the seat-belt of a shared cluster. Set a default; override only where justified. Client-id is hygiene; user is isolation.
Open Scenario 6.2

Lab 07Security

Authentication (who are you?), authorization (what may you do?), encryption + cert-based auth (how the wire is protected). Each scenario stacks on the previous.
7.1

SASL/SCRAM authentication

security

Problem: Default cluster has no authentication — any TCP client that can reach the brokers can produce and consume. SASL/SCRAM adds password-based auth without external dependencies (unlike Kerberos or OAuth).

Demo: add a SASL_PLAINTEXT listener on 9192/9193/9194 alongside the plaintext one; create SCRAM users; connect with and without JAAS config. Failure fingerprint is a distinctive log line.

Two listeners, one broker restart. SASL is add-on, not rip-and-replace — clients migrate at their own pace.
Open Scenario 7.1
7.2

ACLs (what may you do?)

security

Problem: With SASL enabled, principals are authenticated but have no restrictions until ACLs are applied. Common trap: granting Write on topic but forgetting Read on consumer group → consumer stuck at poll().

Demo: enable AclAuthorizer; watch alice denied on unauthorised topics; grant minimum ACLs (Write + IdempotentWrite + Group Read); consume + produce successfully.

SASL says who. ACL says what. Deny-by-default, prefix-scoped, log every deny.
Open Scenario 7.2
7.3

mTLS (encryption + cert-based auth on the wire)

security

Problem: SASL/PLAINTEXT hashes passwords but leaves the payload readable. mTLS gives you encryption AND client identity via certificate — but the setup involves CA, keystores, truststores, and hostname-verification traps.

Demo: generate CA + broker + client certs with openssl and keytool; add SSL listener on 9292/9293/9294; prove that a client without a keystore is rejected at TLS handshake (bad_certificate).

Encryption + identity in one protocol. Client cert is authoritative — TLS handshake fails before any Kafka request is sent.
Open Scenario 7.3

Lab 08KRaft Mode & Migration

Kafka without ZooKeeper. Stand up a KRaft cluster, then walk through the KIP-866 dual-write migration from ZK to KRaft.
8.1

Stand up a KRaft cluster

infrastructure

Problem: ZooKeeper is deprecated (removed in Kafka 4.0). Every 3.x cluster on ZK needs to migrate. First step: understand KRaft.

Demo: parallel kafka-lab-kraft/ folder; 3 combined-mode nodes on 19092–19094 with controllers on 9095–9097. Format storage with cluster UUID. Watch Raft election. Kill the Raft leader — sub-second failover.

KRaft moves cluster metadata into a Raft-replicated Kafka topic. Same broker semantics; simpler operations; entire classes of incidents (stale znodes, session timeouts) disappear.
Open Scenario 8.1
8.2

ZK → KRaft migration (dual-write + cutover)

infrastructure

Problem: Real production clusters can't wipe-and-rebuild. KIP-866 provides a dual-write path: brokers write to both ZK and KRaft during migration, then cut over.

Demo: 5 explicit phases (Prep → Provision → Dual-write → Sync → Cutover). Watch the MigrationState JMX metric advance PRE_MIGRATION → MIGRATION → POST_MIGRATION. One-way cutover.

Migration = phases, not steps. The metric drives every decision. Cutover is one-way — do not rush.

Lab 09Cross-Cluster Replication (MM2)

MirrorMaker 2 for active-passive replication + consumer failover with offset translation. The DR pattern every Ops team eventually inherits.
9.1

MM2 active-passive replication

DR

Problem: Copying data from one cluster to another sounds simple, gets complicated fast — topic prefixing, offset semantics, four internal connectors, replication lag.

Demo: spin up second cluster in kafka-lab-dr/ on 29092–29094. Configure MM2 as standalone process. Produce to source → watch primary.orderevents-2025 appear on target with prefix.

MM2 is async, prefix-by-default, and does not fail over automatically. Set it up, monitor replication-latency-ms, know your RPO.
Open Scenario 9.1
9.2

Consumer failover with offset translation

DR

Problem: Naive failover fails — source consumer's committed offset 42000 means nothing on the target cluster. Depending on auto.offset.reset, consumer restarts from beginning (duplicates) or end (skips).

Demo: enable sync.group.offsets.enabled=true. Consume on source, commit offsets. MM2 mirrors AND translates. Stop source consumer; start target consumer with same group.id → resumes at translated offset. Prove split-brain protection (MM2 refuses to sync when group is active on target).

Offsets don't survive replication automatically. sync.group.offsets.enabled=true makes them survive; never run the same group on both clusters simultaneously.

Lab 10Tiered Storage (KIP-405)

Store hot segments on broker disk, cold segments in cheap object storage. Same topic, same consumer semantics — tier boundary is invisible.
10.1

Enable tiered storage & watch segments offload

storage

Problem: Long retention (30d, 90d, 1y) means expensive brokers with big disks. Tiered storage splits into local (recent, on disk) and remote (older, in object storage). Consumer reads old offsets → broker fetches from remote transparently.

Demo: configure LocalTieredStorage plugin (uses /tmp/kafka-tiered-storage/ as pseudo-remote). Set local.retention.ms=30000, retention.ms=600000. Produce steadily; watch local segments stay bounded while remote grows.

Local for the hot 1%, remote for the long tail. local.retention.ms sets the boundary; consumer semantics don't change.
Open Scenario 10.1

Lab 11Observability at Scale

Beyond dashboards. Alerting rules for the metrics that actually matter, plus consumer lag exporter for the SLO consumers care about.
11.1

Prometheus alerting + AlertManager

monitoring

Problem: Dashboards tell you what's happening now. Alerts tell you when to wake someone up. Alerting badly = alert fatigue → real ones missed.

Demo: extend Lab-02's compose with AlertManager + webhook-echo. Write rules with for: clauses. Route severity=critical vs warning separately. Trigger with kill -9 on a broker → watch KafkaBrokerDown fire, webhook receives the JSON.

Rules are dumb, routing is policy. Alert on symptoms, use for: clauses, inhibit correlated noise, link a runbook.
Open Scenario 11.1
11.2

Consumer group lag monitoring

monitoring

Problem: Lag is the SLO downstream teams actually care about. Broker JMX doesn't give it directly — need a sidecar. Naive lag alert (lag > N) misses the "consumer is dead" case; combined alert covers both.

Demo: add kafka-exporter to Lab-02's compose. Import Consumer Lag dashboard. Start consumer + producer; SIGSTOP the consumer to simulate a stall; watch lag climb; SIGCONT to catch up.

Consumer lag is the SLO consumers actually care about. A single exporter, a single dashboard, two alerts — 90% of downstream incidents visible.

Lab 12Topics · Deep-Dive from the Admin View

The topic is the unit an admin actually operates on. This chapter goes deep — anatomy on disk, config precedence, partition sizing & growth trade-offs, message-size handshake, naming standards, production creation checklist.
Slide decks for this lab: Main deep-dive (82 slides) Config reference (40 slides) Trainer speaker notes
12.1

Topic anatomy on disk (segments, indexes, offsets)

deep

Problem: Every real incident eventually forces an admin to look inside a partition directory. What's a segment? An index? What does kafka-dump-log.sh output actually mean?

Demo: create a topic with small segments; produce until several roll; dissect the partition dir. Walk through .log, .index, .timeindex, partition.metadata, leader-epoch-checkpoint. Use kafka-dump-log.sh to read a record batch.

Every "look inside" tool goes through kafka-dump-log.sh. Rolled segments only; active segment is immune. Never touch log files by hand while broker is running.
Open Scenario 12.1
12.2

Config precedence (broker default vs topic override vs client)

deep

Problem: When a config value doesn't behave as expected, the debugging question is "where did it come from?" Four sources: dynamic topic override, dynamic broker, static broker, code default.

Demo: two topics (inherited vs override). Set cluster-wide broker default dynamically; watch inherited topic pick it up, override topic ignore it. Use --describe --all to see the synonyms= source chain. Audit script to find drift.

Topic override wins over broker dynamic wins over broker static wins over code default. Use --describe --all — the synonyms= list is the debugging story.
Open Scenario 12.2
12.3

Partition sizing & growth (increase allowed, decrease impossible)

deep

Problem: Adding partitions changes hash(key) % N — future keys may map to different partitions than historical ones. Per-key ordering breaks across the change. You cannot decrease.

Demo: produce keyed messages, note partition assignment. Grow 3→6 partitions. Produce same keys again — watch some land on new partitions. Try to shrink → Number of partitions cannot be reduced error.

Adding partitions is a one-way, application-visible change. Size correctly at creation; add later only with the app team's agreement.
Open Scenario 12.3
12.4

Message size limits & auto-create traps

deep

Problem: Message size is a five-layer handshake (producer / broker / topic / replication / consumer). Change one, forget the others → cascading errors. Auto-create silently produces topics with RF=1, 1 partition.

Demo: Part A — walk the size cascade with distinct error messages at each layer. Part B — reproduce the "typo topic" that auto-creates with terrible defaults; then disable auto.create.topics.enable.

Message size is a five-way handshake — strictest wins. Auto-create is convenient in dev and radioactive in prod — turn it off, enforce with ACLs, provision through automation.
